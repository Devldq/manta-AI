## 一、Harness Engineering 核心概念

### 1.1 什么是 Harness Engineering？

**Harness Engineering**（驾驭工程）是 OpenAI 内部团队提出的一种全新工程范式。其核心实验如下：

> **用 5 个月时间，由 3 名工程师驱动 Codex 智能体，交付了一个约 100 万行代码的真实软件产品 —— 其中没有一行代码是人工编写的。**

关键数据：
- **1,500+ PRs** 合并，平均每位工程师 3.5 PR/天
- **效率提升约 10x** —— 仅用手工编码所需 1/10 的时间完成
- 智能体单次运行可持续工作 **6小时以上**（通常在人类睡眠时间）
- 团队扩大至 7 人后，吞吐量还在持续**增加**

### 1.2 核心哲学

```
人类掌舵。智能体执行。
```

工程师角色的根本转变：

| 传统工程师 | Harness 工程师 |
|-----------|--------------|
| 编写代码 | 设计环境与约束 |
| 审查逻辑 | 明确意图与验收标准 |
| 手动调试 | 构建反馈回路 |
| 技术债清理 | 编码"黄金原则" |

## 二、五大核心设计原则

### 原则 1：环境即产品（The Environment is the Product）

> 早期进展慢，不是因为 Codex 能力不足，而是因为**环境规范不够明确**。

智能体所需的三类基础设施：
- **工具（Tools）**：标准开发工具、git worktree、Chrome DevTools 协议
- **抽象层（Abstractions）**：领域分层、固定架构边界、类型化 SDK
- **内部结构（Internal Structure）**：清晰的代码仓库结构、命名规范、设计文档

### 原则 2：上下文是稀缺资源（Context is a Scarce Resource）

> 给 Codex 的是一张地图，而不是一本 1,000 页的说明书。

`AGENTS.md` 反模式 vs 正确做法：

```
❌ 反模式：AGENTS.md 当百科全书
   - 挤占任务和代码的 Token 空间
   - 过多规则导致智能体无法导航
   - 规则迅速腐烂、难以验证

✅ 正确做法：AGENTS.md 当目录（~100行）
   - 短小精悍，指向 docs/ 深层文档
   - 渐进式披露（Progressive Disclosure）
   - CI 自动验证文档的新鲜度和结构正确性
```

### 原则 3：强制不变量（Enforce Invariants）

> 通过强制执行不变量，而非对实施过程进行微观管理，令智能体能够快速交付。

分层架构的单向依赖规则：
```
Types → Config → Repo → Service → Runtime → UI
```
横切关注点通过唯一入口 **Providers** 注入（认证、连接器、遥测、功能标志）。

自动化执行机制：
- 自定义 Linter（由 Codex 生成）
- 结构测试
- CI 门控（合并前自动验证）

### 原则 4：可观测性对智能体开放（Observable by Agent）

> 将应用程序的 UI、日志和指标直接对智能体开放。

具体措施：
- **git worktree 隔离**：每个任务启动独立应用实例，任务完成后销毁
- **Chrome DevTools 协议接入**：DOM 快照、截图、导航 → 智能体可自主验证 UI
- **临时可观测性堆栈**：LogQL 查询日志 + PromQL 查询指标
- 允许如"确保服务启动在 800ms 内完成"这类性能提示成为可执行约束

### 原则 5：黄金原则 + 循环清理（Golden Rules + Garbage Collection）

> 技术债像高息贷款，持续小额偿还比一次性大额偿还要好。

**黄金原则示例：**
1. 优先使用共享工具包，而非手写 helper
2. 在边界处解析数据形状（Parse, don't validate）
3. 不使用"YOLO 式"探测数据，必须依赖类型化 SDK

**循环清理机制：**
- `doc-gardening` 智能体定期扫描陈旧文档，发起修复 PR
- 后台 Codex 任务扫描代码偏差，发起重构 PR
- 大多数清理 PR 可在一分钟内完成审查并自动合并

## 三、Agent 技术架构设计

### 3.1 整体架构图

```
┌─────────────────────────────────────────────────────────────┐
│                      人类工程师层                             │
│  [ 意图定义 ] → [ 任务拆解 ] → [ 验收标准 ] → [ 结果验证 ]    │
└───────────────────────┬─────────────────────────────────────┘
                        │ Prompt
                        ▼
┌─────────────────────────────────────────────────────────────┐
│                      Agent 编排层                             │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                 │
│  │ 任务规划  │   │ 上下文管理│   │ 工具调度  │                 │
│  └──────────┘   └──────────┘   └──────────┘                 │
└───────────────────────┬─────────────────────────────────────┘
                        │
          ┌─────────────┼──────────────┐
          ▼             ▼              ▼
┌──────────────┐ ┌────────────┐ ┌────────────────┐
│   工具层      │ │  知识库层   │ │   可观测性层    │
│ git / gh     │ │ docs/      │ │ Logs (LogQL)   │
│ CLI scripts  │ │ AGENTS.md  │ │ Metrics(PromQL)│
│ CDP browser  │ │ plans/     │ │ Traces (OTEL)  │
│ linter       │ │ design/    │ │ Screenshots    │
└──────────────┘ └────────────┘ └────────────────┘
          │             │              │
          └─────────────┼──────────────┘
                        ▼
┌─────────────────────────────────────────────────────────────┐
│                     应用代码库层                               │
│   Types → Config → Repo → Service → Runtime → UI            │
│                    ↑ Providers（横切关注点）                   │
└─────────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│                     CI/CD 反馈层                               │
│  [ Lint ] → [ Tests ] → [ Build ] → [ PR Review ] → [ Merge ]│
│                ↑ 智能体对智能体审查循环                         │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 知识库结构设计

```
repo/
├── AGENTS.md              # 目录地图 (~100行)，指向 docs/
├── docs/
│   ├── ARCHITECTURE.md    # 域和包分层顶层地图
│   ├── design/            # 设计文档（含验证状态）
│   │   ├── index.md       # 设计文档目录 + 质量评分
│   │   └── {feature}.md   # 各功能设计文档
│   ├── principles.md      # 核心理念 + 黄金原则
│   ├── quality.md         # 文档质量评分矩阵
│   └── tech-debt.md       # 已知技术债（版本化）
├── plans/
│   ├── active/            # 活跃执行计划
│   ├── completed/         # 已完成计划（历史）
│   └── template.md        # 执行计划模板
└── .github/
    └── workflows/
        ├── lint-docs.yml   # 文档新鲜度 CI 检查
        └── agent-review.yml # 智能体审查工作流
```

### 3.3 Agent 任务执行流程

```
人类输入 Prompt
       │
       ▼
┌─────────────────┐
│  1. 读取 AGENTS.md │ ← 获取代码仓库地图
└────────┬────────┘
         │
         ▼

└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  3. 验证代码库状态 │ ← 运行 lint / 结构测试
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  4. 启动隔离环境  │ ← git worktree + 临时可观测性
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  5. 实现任务     │ ← 遵循架构约束和黄金原则
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  6. 本地验证     │ ← 运行应用 + CDP 截图 + 指标检查
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  7. 开 PR + 自审  │ ← 智能体对智能体审查循环

         │
    ┌────┴────┐

    └────┬────┘
```

### 3.4 架构分层约束模型

```
业务域内的严格单向依赖：

  ┌─────────┐
  │  Types  │ ← 最底层，纯数据类型定义，无依赖
  └────┬────┘
       │
  ┌────▼────┐
  │  Config │ ← 配置层，仅依赖 Types
  └────┬────┘
       │
  ┌────▼────┐
  │   Repo  │ ← 数据访问层，仅依赖 Types + Config
  └────┬────┘
       │
  ┌────▼────┐
  │ Service │ ← 业务逻辑层，仅依赖 Repo 及以下
  └────┬────┘
       │
  ┌────▼────┐
  │ Runtime │ ← 运行时层，仅依赖 Service 及以下
  └────┬────┘
       │
  ┌────▼────┐
  │    UI   │ ← 最顶层，仅依赖 Runtime 及以下
  └─────────┘

横切关注点（通过 Providers 统一接入）：
  Auth | Connectors | Telemetry | Feature Flags
```

| 步骤 | 能力 |

## 五、ARM 项目落地建议

基于 Harness Engineering 理念，结合本项目（ARM Console）的落地路径：

### 5.1 近期行动

1. **完善 `AGENTS.md`**：当前不存在，建议创建约 100 行的目录文件，指向各域文档
2. **建立 `docs/` 知识库**：包含架构文档、设计文档索引、黄金原则
3. **增加 Linter 规则**：将"黄金原则"编码为可自动执行的 lint 规则
4. **结构化执行计划**：将 `plans/` 目录用于复杂任务的执行计划，版本控制

### 5.2 中期行动

1. **智能体可观测性**：将日志和指标 API 化，允许 Agent 直接查询应用状态
2. **自动化 PR 审查**：建立智能体对智能体审查工作流（`agent-review.yml`）
3. **doc-gardening 机制**：定期运行文档新鲜度检查，自动发起修复 PR
4. **分层架构强制**：在当前 Next.js 项目中实施 `Types → Service → UI` 单向依赖 lint

### 5.3 长期目标

```
目标：每名工程师 3.5+ PR/天的 Agent 辅助吞吐量
关键：减少人类判断介入点 → 更多决策编码为约束和原则
验收：Agent 可端到端驱动 ARM 功能从 Issue 到 Merge
```

## 六、关键反模式警示

| 反模式 | 症状 | 解决方案 |
|-------|------|---------|
| 指令大爆炸 | AGENTS.md > 500 行 | 拆分到 docs/，保持 ~100 行 |
| 外部知识孤岛 | 设计决策在 Slack/Google Docs | 将所有关键决策提交到仓库 |
| 无边界架构 | 循环依赖、跨层直接调用 | 建立单向依赖 lint 强制执行 |
| 手动代码清理 | 每周人工清理"AI 残渣" | 建立自动化 doc-gardening 循环 |
| YOLO 数据探测 | `(data as any).field` | 在边界处 Parse，使用类型化 SDK |
| 阻塞合并门 | PR 因偶发测试失败积压 | 智能体吞吐量高时，纠错 < 等待成本 |
