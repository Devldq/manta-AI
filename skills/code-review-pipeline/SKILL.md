# 任务目标
执行完整的代码审查流水线，从代码变更获取到生成审查报告。

# 工作流步骤

## Step 1: 获取变更
- 使用 `execute_command` 获取 git diff
- 识别变更文件列表
- 统计变更范围（新增/修改/删除行数）

## Step 2: 静态分析
- 运行 TypeScript 类型检查
- 运行 ESLint 检查
- 收集所有错误和警告

## Step 3: AI 审查
按以下维度逐文件审查变更：
- **安全漏洞**：SQL注入、XSS、敏感信息泄露、路径遍历
- **逻辑错误**：空值处理、边界条件、异步错误处理
- **性能问题**：N+1查询、不必要的重渲染、内存泄漏
- **代码规范**：命名约定、模块结构、类型安全
- **可维护性**：函数复杂度、重复代码、缺少注释

## Step 4: 生成报告
输出结构化的审查报告。

# 输出格式
```markdown
# Code Review Report

## 变更概览
- 分支：[branch-name]
- 变更文件：[N] 个
- 新增行数：[+N] | 删除行数：[-N]

## 静态分析结果
- TypeScript 错误：[N]
- ESLint 错误：[N]
- ESLint 警告：[N]

## AI 审查结果

### 🔴 严重问题 (必须修复)
| 文件 | 行号 | 问题 | 建议 |
|------|------|------|------|
| [file] | [line] | [issue] | [suggestion] |

### 🟡 改进建议 (建议修复)
| 文件 | 行号 | 问题 | 建议 |
|------|------|------|------|

### 🟢 优秀实践 (值得学习)
- [亮点描述]

## 审查结论
- [ ] 通过（可直接合并）
- [ ] 有条件通过（修改后可合并）
- [ ] 不通过（需重新提交审查）

## 审查者
AI Code Review Pipeline v1.0.0
```

# 注意事项
1. 每次审查前确认工作区清洁（无未提交变更）
2. 大变更（>500行）需分批审查
3. 审查意见应具体、可操作，避免主观评价
