# 解决方案总结

## 命令用法

通过 `/solution-summary` 命令调用，支持以下子命令：

- `/solution-summary` - 自动总结（根据触发条件）
- `/solution-summary question` - 专注于总结问题陈述
- `/solution-summary explain` - 专注于解释解决方案和决策
- `/solution-summary summary` - 完整总结（问题+解决方案）

## 触发条件

当满足以下任一条件时，在回复结尾添加总结：

1. **问题复杂度**：问题涉及多个步骤、组件或需要架构决策
2. **调试过程**：调试复杂问题需要深入调查或根本原因分析
3. **重构范围**：代码重构或重大更改影响多个文件或模块
4. **权衡决策**：存在多个可行方案，需要权衡利弊
5. **用户请求**：用户明确要求总结或回顾解决方案

## 指导原则

1. **简洁明了**：总结应简洁，避免冗长细节
2. **突出重点**：强调关键决策和权衡
3. **可操作性**：提供具体的结论和后续步骤
4. **适应性**：根据问题复杂度调整总结详细程度

## 示例

### 示例1：自动总结（调试复杂问题）
```markdown
## 📋 问题总结与解决方案

### 问题陈述
用户登录功能间歇性失败，错误日志显示数据库连接超时。

### 关键挑战
- 问题难以复现，只在特定时间段发生
- 涉及多个服务：前端、认证服务、数据库
- 需要分析生产环境日志

### 采用的解决方案
实施连接池监控和自动重试机制，优化数据库查询索引。

### 关键决策与理由
1. **选择连接池监控而非增加数据库连接数**：因为连接数增加会导致资源耗尽
2. **实施指数退避重试而非立即重试**：避免在高峰期加重数据库负载

### 结果与影响
- 登录失败率从5%降至0.1%
- 数据库负载降低30%
- 需要持续监控连接池使用情况
```

### 示例2：架构重构
```markdown
## 📋 问题总结与解决方案

### 问题陈述
现有单体应用难以维护，需要拆分为微服务架构。

### 关键挑战
- 需要保持向后兼容性
- 数据一致性要求高
- 团队对微服务经验有限

### 采用的解决方案
采用渐进式拆分策略，从最独立的模块开始，使用API网关管理服务间通信。

### 关键决策与理由
1. **选择渐进式拆分而非一次性重构**：降低风险，允许团队逐步学习
2. **使用API网关而非直接服务调用**：简化客户端调用，便于监控和限流

### 结果与影响
- 开发效率提升40%
- 系统可扩展性增强
- 需要建立服务监控和日志聚合系统
```

### 示例3：问题总结（question 子命令）
```markdown
## 🔍 问题分析

### 问题描述
用户报告应用在特定设备上崩溃，错误日志显示内存溢出。

### 根本原因
图片加载组件未正确释放内存，导致内存泄漏。

### 影响范围
- 受影响的组件：图片画廊、用户头像上传
- 用户影响：移动端用户频繁崩溃
- 业务影响：用户留存率下降15%

### 约束条件
- 技术约束：需要兼容旧版Android设备
- 业务约束：不能影响现有功能
- 时间约束：需要在2周内修复
```

### 示例4：解决方案解释（explain 子命令）
```markdown
## 💡 解决方案详解

### 方案概述
实施懒加载和内存池管理，优化图片处理流程。

### 技术实现
1. 使用Intersection Observer实现图片懒加载
2. 实现内存池，复用图片对象
3. 添加内存监控和自动清理机制

### 关键决策
1. **选择懒加载而非预加载**：减少初始内存占用
2. **使用内存池而非频繁创建销毁**：降低GC压力

### 优势与风险
- **优势**：内存使用降低60%，崩溃率下降90%
- **风险**：懒加载可能导致图片显示延迟

### 实施步骤
1. 分析现有图片加载流程
2. 实现懒加载组件
3. 开发内存池管理器
4. 集成内存监控
5. 性能测试和优化
```

## 注意事项

1. **不要过度总结**：简单问题不需要总结
2. **保持客观**：基于事实总结，避免主观臆断
3. **保护隐私**：不在总结中包含敏感信息
4. **及时总结**：在问题解决后立即总结，避免遗忘细节
